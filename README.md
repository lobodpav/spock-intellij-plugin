# Spock IntelliJ Plugin

![No longer maintained](doc/Marketplace/00%20No%20longer%20maintained.png)

> **This plugin is no longer maintained.** See the [reasoning](#why-this-plugin-is-no-longer-maintained) below.

Provides support for the amazing [Spock testing framework](https://spockframework.org).

Inspired by the [Spock Framework Enhancements](https://plugins.jetbrains.com/plugin/7114-spock-framework-enhancements) plugin
which is [not actively maintained](https://github.com/cholick/idea-spock-enhancements/blob/master/README.md),
and does not fully function in the newest IntelliJ versions due to breaking API changes.
# Features


- Validation of block names, their order and completeness (e.g. invalid blocks, missing `then` blocks, etc.)
# Installation


In IntelliJ, go to `Settings -> Plugins` and search for `spock` in the Marketplace.

For more details, see the [plugin page](https://plugins.jetbrains.com/plugin/23380-spock-framework-support).

# Development

1. Generate a private key and a signing certificate. Refer to the [official documentation](https://plugins.jetbrains.com/docs/intellij/plugin-signing.html#generate-private-key).
2. Define environment properties.
    ```shell
    export INTELLIJ_COMMUNITY_SOURCES = "A path to checked out GIT source codes of the IntelliJ Community"
    export INTELLIJ_CERTIFICATE_CHAIN = "The content of the certificate"
    export INTELLIJ_PRIVATE_KEY       = "The content of the private key"
    export INTELLIJ_PRIVATE_PASSWORD  = "The password to the private key"
    export INTELLIJ_PUBLISH_TOKEN     = "The token for plugin publishing to JetBrains Marketplace"
    ```
3. Run `gradle runIde` to execute an IntelliJ instance with the plugin installed.
4. Run `gradle buildPlugin` to build the plugin and prepare the distribution ZIP archive.
# Verification


Verifies the plugin, its configuration and binary compatibility.

```shell
gradle verifyPlugin
```

# Signing

```shell
gradle signPlugin
```

# Release

1. Increment the version in [build.gradle.kts](build.gradle.kts)
2. Update
   - [CHANGELOG.md](CHANGELOG.md)
   - [plugin.xml](src/main/resources/META-INF/plugin.xml)
3. Run `op run -- gradle build verifyPlugin signPlugin publishPlugin`

## Security

All the signing credentials (key, certificate, password) are configured
via environment variables (see the [build.gradle.kts](build.gradle.kts) file).

To avoid storing plain text secrets on the filesystem, [1password CLI](https://developer.1password.com/docs/cli)
can be utilised to inject the credentials to the Gradle process.

```shell
op run -- gradle signPlugin
```

## IntelliJ threading rules


Write actions (such as creating an editor for a loaded file in a test) must be executed on the UI thread only.

To perform a write action in a test, wrap your code inside the `Application.invokeAndWait()` closure.
```kotlin
ApplicationManager.getApplication().invokeAndWait {
    WriteCommandAction.runWriteCommandAction(project) {
        // Your code goes here
    }
}
```

See [IntelliJ threading rules](https://plugins.jetbrains.com/docs/intellij/general-threading-rules.html)
document for more details.

## IntelliJ UI tests

For example, rather than testing a custom File creation action by showing a UI dialog and interacting with it,

## Why this plugin is no longer maintained

After looking for a Spock replacement that would let me write tests in Kotlin alongside my production code,
I found that [Kotest](https://kotest.io/) (FunSpec) + [MockK](https://mockk.io/) +
[Kotest assertions](https://kotest.io/docs/assertions/assertions.html) covers everything I needed from Spock.
Having fully migrated, I no longer use Spock, and I also lack the time to maintain this plugin properly.

### Reasons for migrating away from Spock

1. **Pain with testing Kotlin features from Groovy** — Coroutines and suspend functions require `runBlocking`
   workarounds, value classes and inline classes don't work properly, and sealed classes, extension functions,
   and reified generics are awkward or impossible to test from Groovy.
2. **Groovy's lack of type safety and refactoring issues** — Groovy's dynamic nature means many errors are only
   caught at runtime. IntelliJ's support for Groovy refactoring is not on par with Kotlin — renaming, extracting,
   and navigating code is less reliable, making large test suites harder to maintain.
3. **Spring Boot 4 dropped Spock support** — Spring removed Spock integration as Spock does not yet support
   Groovy 5, making it uncertain whether Spock will keep up with the Spring ecosystem going forward.
4. **Same language for code and tests** — Writing tests in Kotlin removes context switching and allows full use
   of the type system, IDE support, and refactoring tools across production and test code.
5. **Lack of time** — Maintaining an IntelliJ plugin requires keeping up with breaking API changes across IDE versions,
   which is a significant ongoing commitment.

## How I made Kotest feel like Spock

With a bit of setup, Kotest's [FunSpec](https://kotest.io/docs/framework/testing-styles.html#fun-spec) style
provides a very similar testing experience to Spock. Here is a side-by-side comparison:

### Spock

```groovy
class AuthenticationSpec extends Specification {

    def ssmCredentialsService = Mock(SsmCredentialsService)
    def provider = new SsmCredentialsAuthenticationProvider(ssmCredentialsService)

    def "Grants the CLIENT authority when credentials match"() {
        given:
        ssmCredentialsService.getCredentials(CLIENT_NAME) >> makeCreds(CLIENT_NAME, CLIENT_KEY)

        when:
        def auth = provider.authenticate(new UsernamePasswordAuthenticationToken(CLIENT_NAME, CLIENT_KEY))

        then:
        auth.principal == CLIENT_NAME
        auth.credentials == CLIENT_KEY
        auth.isAuthenticated()
    }

    def "Fetches both current and previous credentials when key does not match"() {
        given:
        ssmCredentialsService.getCredentials(CLIENT_NAME) >> makeCreds(CLIENT_NAME, "other-key", 2)

        when:
        provider.authenticate(new UsernamePasswordAuthenticationToken(CLIENT_NAME, CLIENT_KEY))

        then:
        thrown(BadCredentialsException)
        2 * ssmCredentialsService.getCredentials(_)
    }

    def "Throws without calling SSM when the client name is invalid"() {
        when:
        provider.authenticate(new UsernamePasswordAuthenticationToken("foo bar", CLIENT_KEY))

        then:
        thrown(BadCredentialsException)
        0 * ssmCredentialsService._
    }

    def "Throws when principal or credentials is not a String"() {
        when:
        provider.authenticate(new UsernamePasswordAuthenticationToken(principal, credentials))

        then:
        thrown(BadCredentialsException)

        where:
        principal    | credentials
        123          | CLIENT_KEY
        CLIENT_NAME  | 123
    }
}
```

### Kotest (FunSpec) + MockK + Kotest assertions

```kotlin
class AuthenticationSpec : FunSpec({

    val ssmCredentialsService = mockk<SsmCredentialsService>()
    val provider = SsmCredentialsAuthenticationProvider(ssmCredentialsService)

    test("Grants the CLIENT authority when credentials match") {
        // given:
        every { ssmCredentialsService.getCredentials(CLIENT_NAME) } returns makeCreds(CLIENT_NAME, CLIENT_KEY)

        // when:
        val auth = provider.authenticate(UsernamePasswordAuthenticationToken(CLIENT_NAME, CLIENT_KEY))

        // then:
        assertSoftly(auth) {
            principal shouldBe CLIENT_NAME
            credentials shouldBe CLIENT_KEY
            isAuthenticated shouldBe true
        }
    }

    test("Fetches both current and previous credentials when key does not match") {
        // given:
        every { ssmCredentialsService.getCredentials(CLIENT_NAME) } returns makeCreds(CLIENT_NAME, "other-key", 2)
        every { ssmCredentialsService.getCredentials(CLIENT_NAME, 1) } returns null

        // when:
        shouldThrow<BadCredentialsException> {
            provider.authenticate(UsernamePasswordAuthenticationToken(CLIENT_NAME, CLIENT_KEY))
        }

        // then:
        verifyCount {
            2 * { ssmCredentialsService.getCredentials(any()) }
        }
    }

    test("Throws without calling SSM when the client name is invalid") {
        // when:
        shouldThrow<BadCredentialsException> {
            provider.authenticate(UsernamePasswordAuthenticationToken("foo bar", CLIENT_KEY))
        }

        // then:
        confirmVerified(ssmCredentialsService)
    }

    // Parameterised tests replace Spock's `where:` block
    context("Throws when principal or credentials is not a String") {
        // @formatter:off
        withTests(
            row(123,          CLIENT_KEY),
            row(CLIENT_NAME,  456),
        ) { (   principal,    credentials) ->
        // @formatter:on
        
            shouldThrow<BadCredentialsException> {
                provider.authenticate(UsernamePasswordAuthenticationToken(principal, credentials))
            }
        }
    }
})
```

### Key differences

| Spock | Kotest equivalent |
|---|---|
| `given:` / `when:` / `then:` blocks | `// given:` / `// when:` / `// then:` comments |
| `Mock()` / `Stub()` | `mockk<T>()` / `every { } returns` |
| `where:` data tables | `withData` / `withTests` with `row()` |
| `thrown(Exception)` | `shouldThrow<Exception> { }` |
| `with(result) { }` | `assertSoftly(result) { }` |
| `>>` for stubbing | `every { } returns` |
| `* _` for interactions | `verifyCount { }` / `confirmVerified()` |
| Implicit assertions | Explicit infix assertions (`shouldBe`, `shouldContainExactly`, etc.) |

### What Spock does better (and why the tradeoff is worth it)

Spock is not without its strengths — there are areas where it remains more concise or readable than Kotest:

1. **More concise assertions** — Spock's implicit `then:` assertions using `==` are shorter than Kotest's
   `shouldBe`, `shouldContainExactly`, etc. The verbosity is the price for type-safe, explicit assertions
   that the compiler can check.
2. **Assertion failure output** — Spock's power assertions display a diagram of each sub-expression's value
   on failure, which is very helpful for debugging. Kotest can achieve the same with the
   [Kotlin power-assert](https://kotlinlang.org/docs/power-assert.html) compiler plugin — you configure
   which assertion functions it instruments:
   ```kotlin
   // build.gradle.kts
   powerAssert {
       functions = listOf("io.kotest.matchers.shouldBe", "kotlin.assert")
   }
   ```
3. **More readable data tables** — Spock's `where:` block with pipe-separated columns is arguably more readable
   than Kotest's `withTests`/`withData` with `row()`. However, Kotest's approach is type-safe — mismatched types
   are caught at compile time rather than failing at runtime with a cryptic Groovy cast error.
4. **Groovy's spread operator** — Spock allows `a*.b*.c` to navigate nested collections, which has no direct
   equivalent in Kotlin. The replacement is `a.map { it.b }.flatMap { it.c }` or
   `a.flatMap { it.b.map { b -> b.c } }`, which is more verbose.
5. **Stubbing syntax** — Spock's `mock.method() >> value` is more compact than MockK's
   `every { mock.method() } returns value`. A minor difference, but it adds up across a large test suite.
6. **Unrolled test names** — Spock's `@Unroll` with `#variable` interpolation in method names is more natural
   than Kotest's `nameFn` parameter. Kotest's `withTests` with typed `row()` helpers mitigate this by generating
   reasonable names automatically.

Despite these ergonomic trade-offs, having compile-time safety, first-class coroutine support, and a single
language across the entire codebase makes Kotest the better choice for Kotlin projects to me.

### Test isolation via `ProjectConfig`

Kotest supports project-wide configuration through a `ProjectConfig` class. This is where you can set a global
test timeout, register extensions, and — importantly — ensure that MockK state is reset between tests:

```kotlin
object ProjectConfig : AbstractProjectConfig() {

    override val timeout = 15.seconds

    override val extensions: List<Extension> = listOf(
        ClearAllMocksTestListener,
    )
}
```

Where `ClearAllMocksTestListener` calls `clearAllMocks()` after each test to prevent mock state from leaking
between tests:
```kotlin
object ClearAllMocksTestListener : AfterEachListener {
    override suspend fun afterEach(testCase: TestCase, result: TestResult) {
        clearAllMocks()
    }
}
```

This gives you reliable test isolation similar to what Spock provides out of the box, where each feature method
gets a fresh instance of the Specification class.
